### Incorrect
- The analyzer computes Vout = Vref × (1 + R4/R5) = 0.6 × (1 + 61900/2150) = 17.87V and flags a 25.5% mismatch against the +24V net name. This is reported as a design observation. However, the TLV61046ADB datasheet specifies Vref = 0.6V, and with R4=61.9k and R5=2.15k the math is correct. The actual intended output voltage is likely 24V, which would require Vref × (1 + R4/R5) = 24V → Vref = 24 / 29.79 ≈ 0.806V, not 0.6V. The more likely explanation is that the resistor values in the schematic are set for ~18V output (the TLV61046ADB has a 28V max output) and the net is aspirationally named '+24V' by the designer, or the feedback divider values are wrong. The analyzer correctly computes and flags this discrepancy but uses 'heuristic' for Vref source, which is appropriate. The mismatch flag is a genuine finding.
  (signal_analysis)
- The LPUART2_TX net in the JSON output lists three pins: J1 pin 5, U2 pin 30 (PB6), and U2 pin 31 (PB7). In a UART interface, TX and RX should be separate nets. The schematic has two labels both named 'LPUART2_TX' — one at position (267.97, 156.21) connecting to J1 pin 5, and one at position (148.59, 58.42) connecting to the STM32 side. If there is a second STM32 label intended to be 'LPUART2_RX' (PB7, pin 31), it has been given the same net name 'LPUART2_TX' in the schematic, causing both pins to appear in the same net. The analyzer faithfully reports this connectivity but does not flag it as an ERC-style issue (both TX and RX shorted). The UART bus analysis reports U2 twice (devices: [U2, U2]) with pin_count=3, which is anomalous — there is no external UART device. This is a schematic design issue faithfully reproduced, not an analyzer error.
  (signal_analysis)
- The schematic contains P1 (EL-210-KEY) with 3 column outputs (KC0/KC1/KC2) driven from U2 PA0–PA2 and 9 row inputs (KR0–KR8) read by U2. This is a classic keyboard matrix (3 columns × 9 rows = up to 27 keys). However, the analyzer's signal_analysis.key_matrices array is empty []. The key matrix detector apparently does not recognize the EL-210-KEY custom connector symbol as a keyboard matrix since it lacks the standard library pattern. This is a missed detection.
  (signal_analysis)
- The power regulator entry for U4 shows input_rail: null (absent from the output). The TPS613221ADBV has VI/SW (pin 1) connected to L1 (inductor) and its GND (pin 2) on GND. The input rail should be +BATT, as L1 pin 2 is on +BATT. The analyzer detects U6's input_rail as +BATT correctly (via VI pin), but misses U4's input rail. This is likely because the TPS613221ADBV SW pin is typed 'power_in' in the lib_symbol rather than the VI pin being explicitly labeled — TPS613221ADBV's pin 6 in the lib is 'VI' but may not be present in this symbol's pin definition (the symbol only has pins 1=SW, 2=GND, 3=NC, 4=VOUT, 5=NC). The TPS613221ADBV does not have a separate VI pin in this symbol, so the analyzer cannot trace the input supply.
  (signal_analysis)
- The subcircuits array contains two separate entries for U2 (STM32G0B1KET6): one with zero neighbors and one with 7 neighbors (C2, J1, P1, R1, U3, U5, U6). This duplication arises because U2 has two symbol units in the schematic (unit 1 with GPIO pins and unit 2 with VDD/VSS). The analyzer creates a subcircuit entry per unit instance rather than merging them. This is a known multi-unit symbol handling issue.
  (signal_analysis)
