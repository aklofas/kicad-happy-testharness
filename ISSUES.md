# Issue Tracker

Tracker for kicad-happy analyzer bugs (KH-*) and test harness improvements (TH-*).
Contains enough detail to resume work with zero conversation history.

> **Protocol**: When fixing issues, remove them from this file and add to FIXED.md in the
> same session. See README.md "Issue tracking protocol" for full details. Closed issues
> with root cause and verification details are in [FIXED.md](FIXED.md).

> **Reporting guidelines for Level 3 subagents**: Root cause descriptions must cite
> specific function names and line numbers, not just file names. When claiming code
> "doesn't check X", trace the actual code path for the repro input and show which line
> returns the wrong result — don't infer from the symptom what the code must be doing.
> Common pitfalls:
> - Code checks the right field but matches the wrong strings (KH-213: checked keywords
>   for `p-channel` but actual keywords contain `PMOS`)
> - Code has the right pattern but wrong format (KH-209: matched `Vnn` but not `nnVn`)
> - Fix exists but callers bypass it (KH-212: KH-153 fix requires `component_type` param
>   that callers don't pass)
> - Transforms are applied but decomposed wrong (KH-207: `compute_pin_positions` runs but
>   matrix→angle extraction is mathematically incorrect)
>
> Include in every report: (1) the function name and line number that produces the wrong
> result, (2) the actual input values from the repro file, (3) what the code returns vs
> what it should return.

## Test Harness Issues

### TH-013 — Flattened project/file names exceed filesystem NAME_MAX, blocking git checkout on some filesystems (HIGH)

**Symptom:** `git clone` and `git checkout` fail with `File name too long` errors on:

- **eCryptfs** (encrypted home, 143-byte limit) — 77 unique components violate, 395 tracked path occurrences. Common on Ubuntu/Mint installs that used the "encrypt my home folder" option at setup. Confirmed on reporter's Linux Mint 22.3 laptop — `mount | grep ecryptfs` showed `ecryptfs_fnek_sig` (filename encryption enabled), which caps cleartext filenames at ~143 bytes because the base64-ish ciphertext has to fit in the underlying 255-byte ext4 NAME_MAX.
- **Plain ext4/xfs/btrfs/APFS** (255-byte NAME_MAX) — 7 unique components violate, 103 tracked path occurrences. All are Cyrillic project directory names in `reference/Exaster/4Labs/` where UTF-8 encoding inflates Ukrainian to ~2 bytes/char, producing 570-byte directory names. Nobody has hit these yet because git alphabetizes and checkout aborts earlier on smaller eCryptfs offenders, but they WILL fail on plain ext4 once an affected user gets past the eCryptfs blockers.

Result: clone "succeeds" in the sense that the pack is fetched, but the working tree is left incomplete. `git status` shows everything under the failed checkout path as `deleted`. Users can't work with the harness until the names are fixed.

**Root cause:** `utils.py:296-302` builds project directory names by flattening the in-repo path with `_` separators and appending the project file stem:

```python
for pdir, stem in sorted(project_dirs.items()):
    if pdir == ".":
        name = stem
    else:
        name = pdir.replace("/", "_").replace("\\", "_") + "_" + stem
```

Two compounding problems:

1. **Stem duplication wart.** KiCad convention puts the project file in a directory whose basename matches the file stem (`foo_rev.A01/foo_rev.A01.kicad_pro`). The flattener joins them, producing `...foo_rev.A01_foo_rev.A01`. This affects **4,989 of 16,766 project directories (29.8%)** across the reference tree — inflating names by ~30-50% with zero information gain.
2. **No length cap.** The flattener has no awareness of filesystem limits. Deeply nested projects, verbose filenames, and non-ASCII (UTF-8 multi-byte) character sets all push past 143/255 byte limits.

Additionally, the flattening logic is **duplicated across 26 source files** (`utils.py`, `regression/findings.py`, `regression/snapshot.py`, `regression/seed.py`, `regression/compare.py`, `regression/promote.py`, `validate/validate_invariants.py`, `validate/validate_outputs.py`, and others). Fixing only `utils.py` won't catch paths generated by other callers — any re-snapshot via a different code path would reintroduce the long names.

**Scope:**

- 77 unique components > 143 bytes (eCryptfs limit)
- 7 unique components > 255 bytes (ext4 NAME_MAX)
- 395 tracked path occurrences > 143 bytes
- 103 tracked path occurrences > 255 bytes
- 4,989 project directories (29.8% of reference tree) with stem duplication
- 26 source files implementing the flatten pattern inline

**Top offending repos (>143 byte paths):**

|------|------:|

**Fix plan:** Detailed work plan in `TODO-th013-filename-length-fix.md` (local-only). Summary:

1. Centralize flattening into `utils.py:project_key(pdir, stem)` and a matching helper for file-side paths
2. Deduplicate trailing stem when `Path(pdir).name == stem` (fixes the 4,989 cosmetic duplicates, saves ~30-50% on affected names)
3. Cap at **143 bytes** (eCryptfs-safe) with SHA1[:10] hash suffix fallback for anything still too long
4. Replace all 26 ad-hoc call sites with the centralized helpers
5. Migration: rename-in-place via a one-shot script using `git mv` to preserve history
6. Targeted re-snapshot of any affected project refs
7. Unit tests for the helpers including boundary cases and Cyrillic input
8. Verification: scan full tracked tree for any remaining >143 byte components; clean-clone test

**Workaround for users:** None practical. Disabling filename encryption on eCryptfs is destructive (requires re-creating the encrypted home). Cloning to an unencrypted location works as a temporary measure.

**Discovered:** 2026-04-09 via user report from Linux Mint 22.3 laptop.

**Scheduled:** Fix 2026-04-10 on `main` (not a worktree, per user).
